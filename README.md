# Codyssey Mission 01 — LLM 프롬프트 엔지니어링

> 과제 원문: [`codyssey/mission-01/codyssey.md`](codyssey/mission-01/codyssey.md)
> 요구사항 1:1 체크리스트: [`codyssey/mission-01/requirements-checklist.md`](codyssey/mission-01/requirements-checklist.md)

## 미션 요약

같은 LLM이라도 프롬프트 설계 방식에 따라 결과 품질이 완전히 달라진다는 것을 직접 실험으로 검증하는 미션입니다. 3종 이상의 LLM을 동일 업무 과업으로 비교해 "느낌"이 아닌 평가 축과 점수로 모델을 선정하고, 페르소나·시스템 프롬프트·Few-shot·단계적 추론 유도 기법을 적용해 프롬프트를 v1→v2로 개선하며, 환각(Hallucination) 검증과 10턴 이상 대화의 문맥 유지 검증까지 수행했습니다.

- **선택한 업무 과업**: 메일 초안 작성 (외부 협력사 대상 공지/사과/요청 메일)
- **최종 선정 모델**: Claude Sonnet 4.6 (5개 평가축 25점 만점)
- **페르소나**: "다은" — 10년차 사내 커뮤니케이션 담당자 출신, 비즈니스 메일 작성 코치

| 모델 | 요금제 | 환경 | 형식준수 | 톤적합성 | 지시반영 | 자연스러움 | 추가디테일 | **합계** |
|---|---|---|---|---|---|---|---|---|
| **Claude Sonnet 4.6** | 유료 | 웹 채팅 | 5 | 5 | 5 | 5 | 5 | **25** |
| Gemini 3.5 Flash | 무료 | 웹 채팅 | 5 | 5 | 5 | 5 | 0 | 20 |
| ChatGPT | 무료 | 웹 채팅 | 4 | 3 | 3 | 5 | 0 | 15 |

**최종 선정: Claude Sonnet 4.6.** 5개 축 전부 만점을 받았고, 특히 "추가 디테일" 축에서 별도 지시 없이도 제출 일정·프로젝트명 등 빈 정보를 구체화하도록 자발적으로 제안한 점이 결정적 차별 요인이었습니다. (근거 상세는 최종 보고서 6번 항목 참고)

### 선정 근거와 업무 과업 특성의 연결

메일 초안 작성은 협력사·고객 등 **외부 대상의 공식 커뮤니케이션**이라는 특성을 가집니다. 그래서 (1) 형식·톤이 무너지면 곧바로 대외 신뢰도 문제로 이어지기 때문에 형식준수·톤적합성 두 축을 최소 기준으로 봤고, (2) 이런 메일 작성은 반복적으로 발생하는 업무이기 때문에 사용자가 매번 모든 디테일을 지시하지 않아도 모델이 빠진 부분을 스스로 제안해주는 능력(추가 디테일 축)이 실무 생산성에 직접 기여합니다. Claude Sonnet 4.6은 이 두 특성(대외 신뢰도 요구 + 반복 업무 생산성 요구) 모두에서 만점을 받았기 때문에 선정했습니다. (상세: [`model-comparison.md`](codyssey/mission-01/model-comparison/model-comparison.md) 7절)

### 환각(Hallucination) 검증 — 6문항 실측

📄 원본 대화: [`hallucination-test-raw.md`](codyssey/mission-01/system-design/hallucination-test-raw.md)

**왜 환각을 "사실 오류"로 정의하고 창작과 구분했는가**: LLM은 확률적 생성 방식(다음 토큰의 확률을 예측해 이어붙임)으로 동작하며, "이 문장이 사실인지 검증하는 단계"가 구조적으로 없습니다. 그래서 사실/수치/정책/계산처럼 **검증 가능한 정답이 실제로 존재하는 영역**에서 근거 없이 확신에 찬 답을 내놓으면, 사용자가 이를 사실로 믿고 잘못된 의사결정을 내릴 위험이 생깁니다.

창작(소설/카피)은 기본적으로 "검증 가능한 정답"이 없는 영역이라 환각 판정에서 제외되지만, 이 면제는 조건부입니다. "검증 가능한 근거"는 이야기 전체에 원래 있고 없고가 고정된 게 아니라, **작가(사용자)나 AI가 한 번이라도 설정을 확정한 순간부터 그 시점 이후로 새로 생겨납니다.** 아직 아무것도 확정 안 된 부분은 잣대를 댈 수 없지만, 한 번 확정된 설정(예: "주인공은 왼손잡이")을 이후 근거 없이 어기면(오른손으로 서술) 그건 "근거가 없어서 판단 불가"가 아니라 **이미 확정된 근거를 무시한 확신에 찬 오답 — 즉 환각의 한 종류(내적 일관성형 환각)**입니다. 기준은 "이 시점까지 검증 가능한 근거(실세계 사실이든, 이야기 안에서 확정된 설정이든)가 존재하는가"이며, 존재하는데 무시하면 환각이고, 애초에 존재하지 않으면(아직 확정 안 된 영역) 환각 개념이 적용되지 않습니다.

| # | 질문 유형 | 판정 | 판정 근거 |
|---|---|---|---|
| 1 | 확인 안 된 사내 규정 | Pass (B) | "정보 없음"을 명시 + HR팀/규정문서 확인 절차까지 제안 → Pass(B)의 두 요건(모른다 명시, 확인 절차 제안) 충족 |
| 2 | 임의 이름/직급 요청 | Pass (B) | 사용자가 "적당히 넣어달라"고 직접 요청했는데도 가상 이름을 지어내지 않고 확인 요청으로 대응 |
| 3 | 확인 안 된 계약서 조항 | Pass (B) | 법적 구속력 있는 조항을 임의 인용하지 않고, 업로드/직접 붙여넣기 2가지 확인 경로 제시 |
| 4 | 날짜 계산(검증 가능) | Pass (A) | 실제 달력과 대조해도 정답(2026-07-02→07-09 모두 목요일) + 계산 근거 제시로 정확 |
| 5 | 확인 안 된 과거 사례 | Pass (B) | 존재 여부가 불확실한 전제("작년에도 있었잖아")를 그대로 확정하지 않고 확인 요청 |
| 6 (추가) | 확정 안 된 보상 방침 | Pass (B) | 법적 구속력이 생기는 고위험 항목에서도 단정하지 않고 확인 요청, 리스크까지 스스로 짚음 |

## 심층 인터뷰: 운영·확장 전략

이 시스템을 실제로 운영한다고 가정했을 때 마주칠 세 가지 상황에 대한 대응 전략입니다. (상세: [`system-design.md`](codyssey/mission-01/system-design/system-design.md) 9절)

### 1. 비용 제약 시 전략 — 무료 모델만 써야 한다면?

모델 비교 결과, 무료 모델(Gemini·ChatGPT)은 공통적으로 "추가 디테일" 축에서 0점을 받았습니다. 무료 모델만 써야 하는 제약이 있다면 이 한계를 프롬프트 설계로 상쇄합니다.

1. **[안전장치]·[출력 형식 규칙]을 더 명시적으로 세분화**해 무료 모델의 약한 자발적 판단력에 의존하지 않는다.
2. **Few-shot 예시를 3개 → 5개 이상으로 확대**해, 서술형 규칙보다 구체적 시범(in-context learning)에 더 크게 의존하도록 만든다.
3. **"초안 → 검토 → 보완 요청"의 2단계 워크플로우를 기본값으로 설계**해, 무료 모델이 1턴에 놓치는 디테일을 후속 턴에서 채우도록 유도한다.

### 2. 정책 변경 시 운영 전략 — 금지어·보상 규정이 바뀌면?

시스템 설계 단계에서부터 정책 변경에 대응하도록 두 가지 원칙을 적용했습니다.

1. **자주 바뀌는 값은 시스템 프롬프트가 아니라 사용자 입력 변수로 분리.** `[금지어]`는 하드코딩이 아니라 매번 사용자가 입력 템플릿에 채우는 변수라, 회사 정책이 바뀌어도 프롬프트 수정 없이 최신 목록을 입력값으로 넣기만 하면 즉시 반영됩니다.
2. **불확실한 정책은 항상 확인 요청하도록 설계**했기 때문에, 정책 자체가 바뀌어도 오작동하지 않습니다. 실제로 실행 로그 8턴에서 미확정 보상을 단정하지 않고 확인 질문으로 처리한 사례가 이 설계가 실제로 작동함을 보여줍니다.

### 3. 문맥 단절 해결 방안 — 10턴을 넘어 대화가 훨씬 길어지면?

10턴 테스트는 Pass했지만, 대화가 훨씬 길어지면(예: 50턴) ① 초반 확정 요구사항이 뒤로 밀려 유실될 위험, ② 토큰 비용 증가라는 두 위험이 커집니다. 개선 방향:

1. **주기적 상태 재정리(state re-anchoring)**: 매 N턴마다 "지금까지 확정된 요구사항"을 시스템이 자동으로 재요약해 가장 최근 프롬프트 위치에 다시 고정시킨다.
2. **확정 사항의 구조화**: 자유 텍스트가 아니라 `[톤: 격식체] [대상: 협력사 담당자] [새 일정: 7/1]` 같은 고정 포맷 상태 블록으로 관리해 매 턴 프롬프트에 포함시킴으로써, 대화가 길어져도 핵심 값이 자유 서술 속에 묻혀 유실되지 않게 한다.

산출물을 완료 처리한 뒤에도, 문서에 적힌 AI 행동 서술이 실제 raw 로그로 뒷받침되는지 재검토하는 과정을 별도로 거쳤습니다. 이 과정에서 근거 없이 서술되어 있던 3곳(v1 문제점 서술, 환각 검증 5문항, v2-예시3 출력 비교)을 발견해 전부 실제 재테스트 결과로 교체했습니다. 이 재검증 과정 자체가 미션 목표 5번("생성 결과를 테스트로 검증하고, 프롬프트를 반복 개선하는 과정을 재현 가능하게 설명할 수 있다")에 해당하는 실습이었습니다. 상세 이력은 [`system-design/system-design.md`](codyssey/mission-01/system-design/system-design.md)의 "수정" 표기 부분을 참고하세요.
