**A. LLM 원리 & 환각**

1. "확률적으로 생성한다"는 게 정확히 무슨 뜻인가요? 같은 질문에 매번 답이 달라질 수 있다는 뜻인가요?
1번 답변 : ai는 문맥을 통계적으로 학습해 확률적으로 유사한 값을 내줍니다. 만약 ai에게 무엇을 하지, 어떻게 할지 명확한 지시가 없다면, ai 가 명확한 패턴을 학습 하지않기에, 자신이 유추한대로 비슷한 내용으로 답변을 해줄수있기에 같은 질문에 매번 항상 답이 달라진다고 장담하기 어렵지만, 명확한 규칙과 지시가 없다면 매번 같은 질문에 답변이 달라질 확률이 높습니다.
 
2. 환각은 왜 일어나나요? 모델이 일부러 거짓말하는 건가요?
2번 답변  : 모델은 거짓말을 의도하지않습니다. 다만 명확한 정보가 없다면, ai 스스로가 문맥을 유추해서 그럴듯한 답변을 내놓는것이기에 거짓말이라고 보기는 어렵습니다. 환각이 일어나는 이유는 정보가 필요한 세부적인 지시를 했을때 ai 에게 충분한 정보를 주지않을때 일어납니다.

3. 창작(소설/카피)은 왜 환각으로 안 보나요? 기준이 뭔가요?
3번 답변 : 기존 환각은 실존하는 정보에 대한 명확한 수치나 사례가 있기에 환각 사례로 볼수있지만, 창작은 실제하지않는 허구의 정보를 제공 하는것이기에 환각으로 볼수없다. 즉 기준은 이 정보가 실제하는것인지가 기준인것같다
 
4. 다은 시스템 프롬프트에서 환각을 막으려고 구체적으로 어떤 장치를 넣었나요?
4번 답변 : 모호한 답변이나 정확한 수치 또는 사례가 필요한 답변에는 ai가 임의대로 정하지않고 사용자에게 되물어보게끔 설계했습니다.

5. 실측 5문항이 전부 Pass였는데, 만약 안전장치 없는 v1이었다면 어느 지점에서 Fail이 났을까요? (예: 2번 질문에서 이름/직급을 임의로 채웠을 가능성)
5번 답변 : 연기로 인한 보상 프롬프트 단계에서 fail 이 났을것 같다. 이유는 v1 이라면 모호한 정보에 대해 되묻지 않고 임의대로 문맥을 유추해 값을 입력 할 확률이 높기 때문이다.

**B. Zero-shot / Few-shot / 단계적 추론**

1. Zero-shot과 Few-shot 차이가 뭔가요? 이번 과제에서 Few-shot을 어떻게 활용했나요?
1번 답변 : zero shot 은 구체적인 시범 사례를 주지 않은 버전이고 few shot은 구체적인 실행 사례를 준것이다. few shot을 활용하여 ai 에게 무엇을 어떻게 할지 구체적인 사례를 제공하여 패턴을 학습 시켜, 템플릿으로 만들어 활용했다. 덕분에 다른 이메일 양식에도 안전장치와 few shot의 사례로 안정적인 답을 받을 수 있었다.

2. Few-shot 3개 중 왜 하나는 일부러 "모호한 입력"으로 만들었나요?
2번 답변 : ai에게 모호하게 프롬프트를 줬을때 ai가 모호한 정보에 되묻는 사례를 넣음으로써, ai가 임의의 값을 제공 할 확률을 낮추기 위해서 만들었다. 

3. 단계적 추론 유도(v2)가 일반 지시(v1)와 뭐가 다른가요? 실제로 어떤 차이가 났나요?
3번 답변 : 구체적 사례가 없는 v1 지시에는 말의 톤과 금지어의 값을 ? 로 줬는데 되묻지않고 자신이 임의대로 값을 채워서 주었다. v2는 구체적인 프롬프트 값을 주고 톤과 금지어 값을 ?로 주었는데, "정보가 모호하면 다시 되묻는다"라는 안전장치가 작동하여 모호한 값에 대해 ai가 되물어 v1과 다르게 사용자가 요구하는 값을 모두다 충족 시켜주었다.

4. "내부 검토 과정은 최종 답변에 노출하지 마라"는 규칙은 왜 넣었나요? 안 넣으면 어떻게 되나요?
4번 답변 : ai에게 제공한 긴 프롬프트 값을 ai 가 하나하나 사용자에게 확인을 받는다면 빠르게 값을 받을 수 없다고 생각하여, 빠른 값과 가독성을 고려하여 규칙을 넣었다. 해당 규칙을 넣지않는다면 내부 검토 과정을 사용자에게 제공 할 것이고 이로 인해 토큰 사용양도 증가 할것이며, 빠른 값을 곧바로 받을 수 없고 가독성이 떨어질것이다.

**C. 모델 비교 방법론**

1. 평가 축을 왜 이 5개(형식준수/톤/지시반영/자연스러움/추가디테일)로 정했나요? "느낌"이 아니라는 게 무슨 뜻인가요?|
1번 답변 : 문제정의 단계에서 반복적인 업무 이메일 작성을 문제로 정의했었기에 이메일을 쓸때 필수적으로 포함되어야 하는 항목을 5가지로 정했다. 느낌이 아니란것은 ai가 임의대로 유추해서 쓰는것이 아닌, 사용자가 제공하는 프롬프트를 기반으로 임의의 값 없이 업무 수행을 해야 한다는것이다. 

2. 세 모델의 요금제(유료/무료)가 다른데 공정한 비교인가요?|
2번 답변 : 비교 과정에서 cladue 를 유료 버전으로 결제 하였지만, 실상 성능은 sonnet 4.6 작업량 낮음으로 작업 했기 때문에 이 버전은 무료 버전에서도 쓸수있는 기능이기에 공정한 비교라고 생각 하였다. 그럼에도 cladue를 유료 버전으로 사용하고있었기에 형식상 유료 버전이라고 진실성 있게 적었다. 
허나 추가 디테일 부분은 아무래도 유료버전의 영향이 있다 생각한다. 하지만 이 부분이 평가의 큰축을 담당하지않기에 그 항목을 제외하고 형식준수/톤/지시반영/자연스러움 항목으로만 평가를 하더라도 claude가 좋은 성능을 보인것은 변함없다. 

3. "추가 디테일" 축이 없었다면 최종 선정 결론이 바뀌었을까요? (Claude·Gemini 나머지 4축 동점이었던 점과 연결해서 설명)
3변 답변 : 추가 디테일이 없었다면 최종결론은 claude 와 gemini 두 모델을 추천했을것이다.

4. 프롬프트 1회 입력 결과만으로 비교했는데, 재현성/신뢰성 문제는 없나요?
4번 답변 : 1회 입력이지만, 당시 프롬프트에 모호한 정보 없이 구체적 값을 제공했기에 이후 더 테스트를 했어도 비슷한 값을 주었을거라 생각한다.때문에 구체적인 프롬프트 수행을 근거로 재현성과 신뢰성에는 문제가 없다고 판단 할 수 있다.

**D. 페르소나 & 시스템 프롬프트**

1. 페르소나 없이 그냥 "메일 써줘"라고 하면 안 되나요? 페르소나가 결과 품질에 실제로 어떤 영향을 줬나요?
1번 답변 : 페르소나 없이 메일을 썼다면, 무엇을 어떻게 할지 ai에게 구체적 지시를 하지않았기에, ai가 임의대로 판단하여 값을 줬을것이다. 사용자의 의도가 거의 반영되지 않았기에 만족도도 낮았을것이고, 안전장치도 없기에 톤과 금지어 같은 세부적인 사항에서 누락됐을 가능성이 크다. 페르소나를 사용했을때는 사용자가 요구하는 내용을 누락없이 반영 해주었으며, 모호한 답변에는 다시 되물어 세부적인 사항도 챙길수있었다. 전체적으로 결과 품질이 상향되었다.
 
2. "정확성 > 정중함 > 간결함" 우선순위를 왜 이 순서로 정했나요?
2번 답변 : 최우선순위는 사용자의 요구사항이 누락되지 않는것이기에 정확성을 1순위로 정하였고, 이후 세부적인 디테일을 반영하기위해 정중함이 중요하다 생각하였고 세번째로는 메일 가독성을 위해 간결함을 3순위로 정하였다.
 
3. "안전장치"와 "사실/정책/수치 처리 규칙"은 어떻게 다른가요? 겹치지 않나요?
3번 답변 : 안정장치는 사용자가 요구하는 정보를 누락 하지않도록 방지하는 장치이며, 사실/정책/수지 처리 규칙은 임의의 값을 채워넣는 환각을 방지하기위해 세운 규칙이다. 하지만 두 규칙은 제공하지 않는 값에 대한 정보에는 임의대로 채워 넣지않는다는 비슷한 역할을 수행한다.

**E. 검증·개선 & 10턴 대화**

1. 10턴 대화에 "톤 변경"과 "늦은 정보 제공"을 넣었는데, 각각 무엇을 검증하려던 건가요?
1번 답변 : 톤 변경과을 해도 내용 문맥이 잘 유지되는지 검증하기 위해 톤 변경을 진행했고, 늦은 정보 제공 또한 문맥 유지를 확인하기 위함이였다.

2. 6턴 보안 사고 건에서 완곡한 표현을 제안한 건 환각 방지인가요, 안전장치인가요? (execution-log.md에서 이미 "업무 판단/안전장치 영역"으로 구분해둔 이유를 설명할 수 있어야 함)
2번 답변 : 정확한 수치와 값이 필요한 정보였다면 환각방지로 봐도됬겠지만, 해당건은 ai가 회사의 민감한 보안사고를 그대로 반영하지않고 업무차원에서 사용자에게 대안을 제시했기 때문에 안정장치 시점으로 봐야한다.

3. 8턴 보상 약속 건은 왜 환각 Fail 기준에 해당할 뻔한 케이스였나요?
3번 답변 : 보상에 대한 수치와 범위를 제공하지않았다. v1 이였다면, 환각이 발생했을떄 ai 가 임의대로 값을 채워넣었을 가능성이 컸기에 fail 기준에 해당 했을것같다.
 
4. "문맥 유지 Pass" 판단 기준이 뭔가요? 그냥 이전 내용을 기억만 하면 Pass인가요?
4번 답변 :  이후 정보를 추가하여도 사용자가 입력한 필수 포함 항목들을 누락하지않고 문맥이 유지되는것을 기준으로 pass로 판단하였다.
