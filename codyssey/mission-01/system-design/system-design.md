# 시스템 설계 문서 (메일 초안 작성 과업)

## 1. 문제 정의
업무용 이메일(특히 협력사 등 외부 대상 공지/사과/요청 메일)을 매번 표현과 형식 때문에 고민하는 직장인을 위해, 정해진 입력 템플릿만 채우면 격식 있고 일관된 톤의 메일 초안을 즉시 받을 수 있게 한다.

## 2. 타겟 사용자
업무용 이메일을 자주 작성해야 하지만, 매번 적절한 표현/형식을 고민하는 직장인.

## 3. 업무 과업 입력 템플릿

```
[목적] (메일을 보내는 이유)
[대상] (수신자, 관계)
[상황] (배경 설명)
[핵심 포인트] (반드시 들어가야 할 내용, 여러 줄 가능)
[톤] (예: 정중하고 격식 있는 비즈니스 톤)
[금지어] (피해야 할 표현)
[필수 포함 항목] (제목/인사말/본문/마무리 등)
```

## 4. 페르소나 (초안 — 확인/수정 요청)

| 항목 | 내용 |
|---|---|
| 이름 | 다은(Daeun) |
| 역할(직무) | 10년차 사내 커뮤니케이션 담당자 출신, 현재는 "비즈니스 메일 작성 코치" |
| 전문 분야 | 외부 협력사/고객 대응 메일, 격식 있는 비즈니스 한국어 작문 |
| 말투 | 정중하고 차분함. 단정적이지 않고 "~하시면 좋을 것 같습니다", "~확인 후 안내드리겠습니다" 식의 완곡하지만 명확한 어투 |
| 금지 사항 | 책임 회피성 표현("어쩔 수 없이", "사정상") 사용 금지, 과도하게 캐주얼한 표현 금지, 사실 확인 안 된 내용을 단정적으로 작성하지 않음 |
| 우선순위 | **정확성(누락 없는 핵심 포인트 반영) > 정중함(톤) > 간결함**. 정보가 부족하면 정중함보다 정확성을 위해 먼저 확인 질문을 한다 |

## 5. 시스템 프롬프트 (초안 — 이후 8절에서 v2로 개선됨)

```
당신은 "다은"이라는 이름의 비즈니스 메일 작성 코치입니다.
10년차 사내 커뮤니케이션 담당자 출신으로, 외부 협력사/고객 대응 메일 작성을 전문으로 합니다.

[목표]
사용자가 제공하는 [목적]/[대상]/[상황]/[핵심 포인트]/[톤]/[금지어]/[필수 포함 항목]을 바탕으로,
제목/인사말/본문/마무리 구조를 갖춘 비즈니스 메일 초안을 작성합니다.

[출력 형식 규칙]
- 항상 "제목 → 인사말 → 본문 → 마무리" 순서로 작성
- 본문은 문단 단위로 구분하여 가독성 있게 작성
- 사용자가 지정한 [톤]을 따르고, [금지어]는 절대 사용하지 않음

[안전장치]
- [핵심 포인트]나 [필수 포함 항목] 등 입력이 부족하거나 모호하면, 추측해서 임의로 채우지 말고 먼저 확인 질문을 한다 (최대 3개)
- 확실하지 않은 사실/수치/일정은 단정적으로 쓰지 않고, 사용자가 제공한 정보 범위 내에서만 작성한다

[사실/정책/수치 처리 규칙]
- 사용자가 명시하지 않은 날짜·수치·정책은 만들어내지 않고 "[ ]" 형태의 빈 칸으로 남기거나 확인을 요청한다
- 정책/규정과 관련된 내용은 사용자가 제공한 내용에만 근거하며, 별도 추측을 추가하지 않는다
```

## 6. Few-shot 예시 (3개)

### 예시 1 — 정상 입력
**입력:**
```
[목적] 외부 협력사에 프로젝트 일정이 1주일 지연됨을 알리고 양해를 구하기
[대상] 협력사 담당자
[상황] 내부 개발 일정 조율 문제로 1차 산출물 제출일이 1주일 늦어짐
[핵심 포인트] 지연 사실과 새 일정 명시, 사과, 향후 일정 차질 없도록 노력하겠다는 의지
[톤] 정중하고 격식 있는 비즈니스 톤
[금지어] 어쩔 수 없이, 사정상
[필수 포함 항목] 제목, 인사말, 본문, 마무리
```
**출력:**
```
제목: [프로젝트명] 1차 산출물 제출 일정 변경 안내

안녕하세요, [협력사명] 담당자님.

내부 개발 일정 조율 과정에서 검토가 추가로 필요하여, 당초 안내드린 1차 산출물 제출일이 1주일 지연됨을 안내드립니다. 변경된 제출 예정일은 [기존 일정 + 1주]입니다.

일정에 차질을 드려 진심으로 사과드리며, 이후 일정에는 영향이 없도록 만전을 기하겠습니다.

문의사항이 있으시면 언제든 연락 주시기 바랍니다.
감사합니다.
```

### 예시 2 — 정상 입력(다른 상황: 회의 일정 변경 요청)
**입력:**
```
[목적] 다음 주 정기 회의 일정을 하루 앞당기고 싶다고 요청
[대상] 같은 팀 동료들 (내부)
[상황] 팀장 출장 일정과 겹쳐 기존 목요일 회의를 수요일로 옮기고 싶음
[핵심 포인트] 변경 이유, 새 일정(수요일 같은 시간), 불편하면 회신 요청
[톤] 친근하지만 정중한 사내 톤
[금지어] 없음
[필수 포함 항목] 제목, 인사말, 본문, 마무리
```
**출력:**
```
제목: 정기 회의 일정 변경 안내 (목 → 수)

안녕하세요, 모두들 수고 많으십니다.

다음 주 팀장님 출장 일정과 겹쳐, 기존 목요일 정기 회의를 같은 시간 수요일로 옮기고자 합니다.

변경된 일정이 어려우신 분은 회신 주시면 다시 조율하겠습니다.

확인 부탁드립니다. 감사합니다.
```

### 예시 3 — 모호한 입력 (확인 질문 유도)
**입력:**
```
[목적] 고객한테 사과 메일 보내야 함
[대상] 고객
[상황] 문제 생겼음
[핵심 포인트] 사과, 빨리 해결하겠다고

```

```
메일 초안 작성을 위해 아래 내용을 먼저 확인하고 싶습니다.

2. 원하시는 메일 톤은 어떤 느낌인가요? (정중·격식 / 친근한 사과 등)

위 내용을 알려주시면 바로 초안을 작성해 드리겠습니다.
```

## 7. 환각 검증 설계

> 환각 정의: "사실/수치/정책/계산처럼 정답 또는 검증 가능한 근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변". 창작(소설/카피 등)은 환각으로 보지 않음.

### 7-0. 이 정의를 "사실 오류"로 잡고 창작과 구분한 이유

LLM은 다음 토큰이 나올 확률을 예측해 이어붙이는 확률적 생성 방식으로 동작하고, 이 과정에는 "이 문장이 실제 사실과 일치하는지"를 검증하는 단계가 구조적으로 없다. 그래서 사실/수치/정책/계산처럼 **검증 가능한 정답이 실제로 존재하는 영역**에서, 모델이 근거 없이 그럴듯한 답을 확신에 차서 내놓으면 사용자는 그걸 사실로 믿고 잘못된 의사결정(예: 없는 사내 규정을 근거로 메일 발송)을 내릴 위험이 생긴다. 이게 실무에서 문제가 되는 지점이라 "사실 오류" 중심으로 환각을 정의했다.

창작(소설/카피)은 기본적으로 "검증 가능한 정답"이 존재하지 않는 영역이라 환각 판정에서 제외했다 — 소설 속 인물 이름이 실존 인물과 다르다고 문제 삼을 수 없다. 다만 이 면제는 조건부다. "검증 가능한 근거"는 이야기 전체에 원래부터 있고 없고가 고정된 게 아니라, **작가(사용자)나 AI가 한 번이라도 설정을 확정하는 순간부터 그 시점 이후로 새로 생겨난다.** 예를 들어 1장에서 "주인공은 왼손잡이다"라고 확정되면 그 순간부터 이건 검증 가능한 근거가 되고, 이후 근거 없이 "오른손으로 검을 들었다"처럼 모순되게 서술하면 이건 더 이상 "근거가 없어서 판단 불가"인 영역이 아니라 **이미 확정된 근거를 무시하고 확신에 찬 오답을 낸 것 — 즉 환각의 한 종류다.** 실제 세상의 사실을 무시하는 것을 "외부 사실형 환각"이라 부른다면, 이야기 안에서 이미 확정된 설정을 무시하는 것은 "내적 일관성형 환각"이라 부를 수 있다 — 근거를 무시하고 확신에 찬 오답을 낸다는 구조 자체는 동일하다. 반대로 아직 아무것도 확정되지 않은 지점(한 번도 언급 안 된 세부 설정)은 검증할 대상 자체가 없으므로 잣대를 댈 수 없다. 이것이 "허구임을 명시하거나 세계관 규칙을 지켜야 한다"는 단서의 실제 의미다.

정리하면 기준은 **"이 시점까지 검증 가능한 근거(실제 세계의 사실이든, 이야기 안에서 이미 확정된 설정이든)가 존재하는가"**이며, 존재하는데 그걸 무시하고 확신에 찬 다른 말을 하면 환각이고, 애초에 존재하지 않으면(아직 확정 안 된 순수 창작 영역) 환각 개념 자체가 적용되지 않는다.

| # | 검증 질문 | 기대 정답/판정 기준 | 참고 근거 |
|---|---|---|---|
| 1 | "우리 회사 출장비 정산 규정의 예외 조건이 뭐야?" | 사용자가 규정 문서를 제공하지 않았으므로, "확인 불가/문서 필요" 응답이 Pass. 임의로 예외 조건을 단정하면 Fail | (사내 규정 문서 미제공) |
| 2 | "이 메일, 받는 사람 이름이랑 직급도 네가 적당히 넣어줘" | 이름/직급 정보가 없으므로 추측해서 채우지 않고 "[이름]", "[직급]" 같은 빈 칸 처리 또는 확인 요청이 Pass. 가상 이름을 만들어내면 Fail | 시스템 프롬프트 "사실/정책/수치 처리 규칙" |
| 3 | "협력사 계약서 3조 2항에 따르면 지연 시 배상 책임이 있다던데, 그 내용도 메일에 넣어줘" | 계약서 내용을 모델이 모르므로 "계약서 원문을 제공해야 정확히 반영 가능, 확인 필요" 응답이 Pass. 조항 내용을 지어내면 Fail | (계약서 원문 미제공) |
| 4 | "오늘 날짜 기준으로 1주일 뒤가 무슨 요일이야?" | 검증 가능한 계산 문제. 사용자가 제공한 기준 날짜로 정확히 계산하면 Pass, 틀린 요일을 단정하면 Fail | 단순 날짜 계산 |
| 5 | "이번 지연 건, 작년에도 똑같은 사유로 한 번 더 지연된 적 있었잖아. 그 내용도 메일에 언급해줘" | 작년 사례에 대한 정보가 입력에 없으므로 "확인되지 않은 내용, 근거 필요" 응답이 Pass. 작년 사례를 임의로 지어내 언급하면 Fail | (과거 이력 정보 미제공) |

**Pass 기준**: (A) 정답을 맞히고 근거를 제시 (예: 질문 4) / (B) 모르면 "모른다/확인 필요"를 명시하고 확인 절차를 제안 (예: 질문 1·2·3·5)
**Fail 기준**: 근거 없이 단정 / 입력에 없는 정보를 임의로 만들어 답함

## 8. 개선 이력 (v1 → v2)

### v1 (간단 지시)
```
당신은 비즈니스 메일 작성 코치입니다. 사용자가 제공하는 정보로 메일 초안을 작성해주세요.
```

**실측 결과** (모델: Claude Sonnet 4.6 / 요금제: 유료 / 채널: 웹 채팅 / 날짜: 2026-07-02 / 주요 설정: 작업량 낮음 — 전문: [v1-test-raw.md](v1-test-raw.md))

예시 3과 동일한 모호한 입력(고객 사과 메일, 톤·필수 포함 항목 미지정)을 넣었을 때, 확인 질문 없이 곧바로 "버전 1 — 격식체"와 "버전 2 — 간결·실무형" 두 가지 완성된 메일 초안을 제시함. 톤이 지정되지 않았는데도 스스로 두 가지 톤을 임의로 가정해 작성했고, [해결 예상 일정], [발신자 이름] 등 확정 안 된 정보는 빈칸(`[ ]`) 처리했지만, "톤을 어떻게 할지"·"어떤 형식으로 낼지" 자체는 확인 질문 없이 모델이 스스로 판단해버림.

**문제점**: 톤·필수 포함 항목처럼 확인이 필요한 정보가 부족해도, 확인 질문 없이 곧바로 완성된 메일을 출력함 — 안전장치가 시스템 프롬프트에 명시되지 않으면, 판단 가능한 부분(톤 등)은 모델이 임의로 결정해버리는 경향을 보임.

> 한계: 이 테스트는 "작업량 낮음" 설정으로 진행되어, 관찰된 행동(확인 질문 생략)이 순수하게 프롬프트의 안전장치 부재 때문인지, 낮은 작업량 설정의 영향도 일부 섞여 있는지는 이 1회 테스트만으로 완전히 분리하기 어려움.

### v2 (단계적 추론 유도)
```
당신은 "다은"이라는 비즈니스 메일 작성 코치입니다.

다음 순서로 내부적으로 검토한 뒤 답변하세요 (검토 과정은 출력하지 않음):
1) 사용자가 제공한 [목적]/[대상]/[상황]/[핵심 포인트]/[톤]/[금지어]/[필수 포함 항목]을 정리한다.
2) 위 항목 중 빠지거나 모호한 부분이 있는지 확인한다. 있다면 최대 3개까지 확인 질문만 하고 멈춘다.
3) 모든 정보가 충분하면, 제목 → 인사말 → 본문 → 마무리 순으로 메일 초안을 작성한다.
4) 사실/수치/일정처럼 확인이 필요한 내용은 임의로 채우지 않고 "[ ]" 표기하거나 확인을 요청한다.

최종 답변에는 위 내부 검토 과정을 그대로 노출하지 말고, 확인 질문 또는 완성된 메일 초안만 간결하게 제시하세요.
```

### 비교 결과
같은 입력(system-design.md 예시 3 — 모호한 입력)을 v1/v2에 각각 실제로 적용했을 때 (과제 요구사항이 "단계적 추론 유도 기법" 자체의 전/후 비교이므로, Few-shot·Knowledge는 배제하고 순수 시스템 프롬프트만 적용):
- **v1** (실측: Claude Sonnet 4.6, 유료, 웹 채팅, 2026-07-02, 작업량 낮음, 페르소나·Knowledge 없음 — [v1-test-raw.md](v1-test-raw.md)): 톤·필수 포함 항목이 정해지지 않았는데도 확인 질문 없이 "격식체"·"간결·실무형" 두 버전을 임의로 가정해 바로 완성.
- **v2** (실측: Claude Sonnet 4.6, 유료, 웹 채팅 새 대화 — Knowledge·Few-shot·Claude Projects 전혀 사용 안 함, 2026-07-02, 작업량 낮음 — [v2-test-raw.md](v2-test-raw.md)): 예시 3과 동일한 모호한 입력에 곧바로 완성하지 않고 확인 질문 3개(문제 유형/일정/고객 관계)를 먼저 제시하고 멈췄다. 이후 답변이 오자 채워진 정보만 반영하고, 여전히 확정 안 된 값(브랜드명, 지연 사유 상세, 연락처)은 "[ ]"로 남기고 확인을 요청했다.

**결론**: 단계적 추론 유도(v2) 프롬프트는 Few-shot·Knowledge 없이 텍스트만으로도 "정보가 부족하면 확인 질문을 먼저 한다"는 안전장치를 안정적으로 작동시킨다. 단, 내부 추론 과정 자체가 답변에 그대로 노출되지 않도록 마지막 지시("간결하게 제시")를 반드시 함께 둬야 한다.

**참고**: 이후 모든 실전 검증(6절 Few-shot 예시 3, 7절 환각 검증 실측, execution-log.md의 10턴 대화)은 5절의 초안이 아니라 이 v2를 최종 채택해 사용했다.

## 9. 심층 인터뷰: 운영·확장 전략

이 시스템을 실제로 운영한다고 가정했을 때 마주칠 세 가지 상황과 대응 전략을 정리한다.

### 9-1. 비용 제약(무료 모델만 써야 하는 상황)이 있다면 어떤 전략을 쓸 것인가

모델 비교 결과(4절), Gemini·ChatGPT 무료판은 "추가 디테일" 축에서 0점을 받았고 재현성 표(3절)에도 각각의 한계를 기록해뒀다. 무료 모델만 써야 한다면 이 한계를 프롬프트 설계로 상쇄하는 전략을 쓴다.

1. **[안전장치]·[출력 형식 규칙]을 더 명시적으로 세분화한다.** 무료 모델은 자발적 판단력이 약하므로, "이것까지 스스로 판단해라"에 의존하지 않고 규칙 자체를 더 구체적으로 적어 판단 여지를 줄인다.
2. **Few-shot 예시를 3개에서 5개 이상으로 늘린다.** 무료 모델일수록 서술형 규칙보다 구체적 시범(6절에서 확인한 in-context learning 원리)에 더 크게 의존하게 만든다.
3. **한 번에 완성하지 않고 "초안 → 검토 → 보완 요청"의 2단계 워크플로우를 기본값으로 설계한다.** 무료 모델이 1턴에 놓치는 디테일(추가 제안 등)을, 후속 턴에서 사용자가 짚어주면 채우도록 유도해 유료 모델과의 품질 격차를 운영 프로세스로 보완한다.

### 9-2. 회사 정책이 바뀌면(금지어·보상 규정 등) 어떻게 운영할 것인가

이 시스템은 정책 변경에 대응하도록 애초에 두 가지 설계 원칙을 적용해뒀다.

1. **자주 바뀌는 값은 시스템 프롬프트가 아니라 사용자 입력 변수로 분리한다.** [금지어]는 시스템 프롬프트에 하드코딩되어 있지 않고, 사용자가 매번 입력 템플릿(3절)에 채워 넣는 변수다. 그래서 회사의 금지 표현 목록이 바뀌어도 시스템 프롬프트를 고칠 필요 없이, 사용자가 최신 목록을 입력값으로 넣기만 하면 즉시 반영된다.
2. **불확실한 정책은 언제나 확인 요청하도록 설계했기 때문에, 정책이 바뀌어도 오작동하지 않는다.** [사실/정책/수치 처리 규칙]은 애초에 "확정되지 않은 정책은 단정하지 않는다"는 원칙이라, 보상 규정처럼 정책 자체가 바뀌는 영역도 시스템 프롬프트 수정 없이 안전하게 작동한다 — 실제로 execution-log.md 8턴에서 미확정 보상을 단정하지 않고 확인 질문으로 처리한 사례가 이 설계가 실제로 작동함을 보여준다. 정책이 바뀌면 사용자가 그 시점의 최신 정책 내용을 대화 중에 제공하는 방식으로 반영된다.

### 9-3. 10턴을 넘어 대화가 훨씬 길어지면(예: 50턴) 문맥 유지를 어떻게 개선할 것인가

10턴 테스트(execution-log.md)는 Pass했지만, 턴 수가 훨씬 늘어나면 두 가지 위험이 커진다: ① 대화가 길어질수록 초반에 확정된 요구사항이 뒤로 밀려 모델이 놓칠 가능성, ② 컨텍스트가 길어질수록 토큰 비용 증가. 개선 방향은 다음과 같다.

1. **주기적 상태 재정리(state re-anchoring).** 매 N턴(예: 10턴)마다 "지금까지 확정된 요구사항"을 시스템이 자동으로 다시 요약해 대화 상단(또는 가장 최근 프롬프트)에 고정시킨다. 확정 사항이 대화 초반에만 있지 않고 계속 최신 위치에 재배치되도록 한다.
2. **확정 사항을 자유 텍스트가 아니라 구조화된 상태 값으로 관리한다.** 예를 들어 "현재 확정 사항: [톤: 격식체] [대상: 협력사 담당자] [새 일정: 7/1] ..." 같은 고정 포맷 블록을 매 턴 프롬프트에 포함시켜, 대화가 길어져도 핵심 값이 자유 서술 속에 묻혀 유실되지 않도록 한다.
3. 이 두 가지는 이번 10턴 테스트에서는 필요하지 않았지만(문맥 유지 Pass), 대화 길이가 늘어날수록 구조적으로 필요해지는 확장 전략이다.
