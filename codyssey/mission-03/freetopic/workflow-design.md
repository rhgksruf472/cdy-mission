# 프로젝트 2 — 자유 주제: Codyssey 학습 기록 자동화

## 1. 반복 업무 정의

Codyssey 미션을 하나 끝낼 때마다 학습 내용을 정리해서 스터디 채널에 공유하고 진행률을 기록하는 작업을, 미션을 마칠 때마다 수작업으로 반복하고 있다. 이 과정을 자동화한다.

## 2. 선정 도구 및 이유

**Make** 선정. 이유:
- 조건 분기(Router)를 시각적 노드로 다루기 쉬워 "보너스 포함 여부" 분기를 직관적으로 설계할 수 있음
- 무료 플랜(월 1,000 Ops) 범위 안에서 개인 학습 기록 자동화 용도로 충분히 운영 가능
- Notion, Google Sheets, Discord/Slack 커넥터를 모두 기본 지원

## 3. 사전 준비물

1. **Notion 데이터베이스** (신규 생성, 기존 학습 기록 허브와는 별개) — 이름: `미션 학습 로그`
   - 속성: `미션명`(제목) / `학습노트`(텍스트, 이번 미션에서 배운 점 자유 서술) / `보너스포함`(체크박스) / `날짜`(날짜)
   - ⚠️ Make/Zapier의 Notion Trigger는 일반 하위 페이지가 아니라 **데이터베이스(표) 형태만 "새 항목" 감지가 가능**하다. 기존 Notion 학습 허브는 하위 페이지 구조라 바로 연동이 안 되므로, 이 자동화 전용 데이터베이스를 별도로 만든다.
   - ✅ **생성 완료(2026-07-09)**: https://app.notion.com/p/69fbd3d066634594ab96f1090acac73e — 4개 속성 그대로 생성. Make Notion 연결 시 이 DB를 선택하면 됨.
2. **Google Sheets** — `진행률` 시트 (컬럼: 미션명 / 날짜 / 보너스포함)
3. **Discord 또는 Slack** — 알림 받을 채널 1개 (스터디 서버 채널 등)
4. **생성형 AI 연동** — Make의 OpenAI(ChatGPT) 모듈 또는 Anthropic(Claude) 모듈 (API Key 필요, 무료 크레딧 범위 내 사용)

## 4. 워크플로우 흐름

```
[Trigger] Notion — "미션 학습 로그" 데이터베이스에 새 항목 추가
    │
    ▼
[Action 1] 생성형 AI(ChatGPT/Claude) — "학습노트" 내용을 한 줄로 요약
    │
    ▼
[Router] "보너스포함" 체크박스 값으로 분기
    │
    ├── 분기 A: 체크됨(보너스 포함)
    │       └─ [Action 2a] Discord/Slack — "🎉 보너스까지 완료!" 축하 메시지 + AI 요약 전송
    │
    └── 분기 B: 미체크(기본 요구사항만)
            └─ [Action 2b] Discord/Slack — "✅ 미션 완료" 일반 메시지 + AI 요약 전송
    │
    ▼
[Action 3] Google Sheets "진행률" 시트에 행 추가 (미션명 / 날짜 / 보너스포함 여부) — 두 분기 공통 실행
```

- Trigger 1개 (Notion 새 항목)
- Action 3개 (AI 요약 / Discord·Slack 알림 / Sheets 기록) — 2개 이상 충족
- 조건 분기 1개 (보너스포함 체크박스)
- 테스트 시 보너스 체크 O/X 항목을 각각 1개씩 추가해 두 분기 모두 실행 결과 확인

이 워크플로우는 **Notion에 새 항목이 추가되는 순간(Trigger) 자동으로 실행**되므로 "Trigger 발생 시 자동 실행" 제약을 충족한다.

## 5. 보너스 1 (AI 연동 Action) 충족 지점

## 7. Make 구현 중 발생한 이슈 (시행착오 기록 — 구현 과정 요약/평가 대비용)

**문제**: Notion 트리거의 "미션명"(제목)·"날짜" 속성을 Discord 메시지, Google Sheets 컬럼에 매핑한 뒤 첫 실행(Run once)했더니, 값이 들어가야 할 자리에 원본 JSON 구조가 그대로 노출됨.
- Sheets 미션명(A) 칸: `{"type":"text","text":{"content":"Mission 01 - LLM 프롬프트 엔지니어링","link":null},"annotations":{"bold":false,...},"plain_text":"Mission 01 - LLM 프롬프트 엔지니어링","href":null}` — 값이 아니라 구조 전체가 텍스트로 삽입됨
- Sheets 날짜(B) 칸: `{"start":"2026-07-...", ...}` — 마찬가지로 객체 전체 노출
- Discord 메시지도 동일 패턴으로 미션명 자리에 JSON 전체 노출
- 대조군: 보너스포함(체크박스)은 같은 방식으로 매핑했는데도 `TRUE`/`FALSE`로 정상 표시됨 — 체크박스는 단순 boolean이라 문제가 없었음
- 참고로 학습노트(Gemini 프롬프트에 매핑)는 같은 배열 구조 문제가 있었을 것으로 추정되지만, AI가 JSON 안에서 의미를 스스로 추출해 요약해준 덕분에 최종 Discord 메시지의 "요약:" 부분은 정상적으로 보임(원인 발견 전까지는 문제를 인지하지 못했던 지점)

**원인 추정**: Notion API는 "제목"·"텍스트" 속성을 서식(굵게/기울임 등) 정보까지 포함한 **배열** 구조로 반환하고, "날짜" 속성은 시작일/종료일을 담은 **객체** 구조로 반환한다. Make 필드 매핑 패널에서 최상위 칩(`미션명[]`, `날짜`)을 그대로 클릭하면 이 구조 전체가 텍스트 필드에 JSON 문자열로 직렬화되어 삽입된다 — 실제로 필요했던 건 그 안의 leaf(단일 값) 노드였다.

**수정**: 필드 매핑 패널에서 검색 후 트리를 펼쳐 하위 항목까지 확인 →
- 미션명: 기존 `Properties Value.미션명[]` 전체 대신 → `Properties Value.미션명[].Text.Content`와 동일한 값을 가진 형제 노드 **`Plain Text`** 선택
- 날짜: 기존 `Properties Value.날짜` 전체 대신 → 하위 **`Start`** 선택

Sheets 분기 A(보너스 O)의 미션명·날짜 컬럼에 우선 적용 → 칩이 `Properties Value.미션명[]: Plain Text`, `Properties Value.날짜: Start` 형태로 바뀜.

**학습 포인트**: 노코드 도구의 필드 매핑 UI는 앱마다 데이터 구조가 다르다는 걸 그대로 드러낸다. 서식을 지원하는 속성(Notion의 제목·텍스트)은 "배열"로, 복합 값(날짜의 시작/종료)은 "객체"로 온다. 배열/객체 형태의 최상위 칩을 통째로 텍스트 필드에 넣으면 값 대신 JSON 원본이 노출될 위험이 있다 — 매핑 직후 반드시 실제 실행 결과(Discord 메시지, Sheets 행)를 눈으로 확인해야 이런 문제를 잡아낼 수 있고, 필드 트리를 펼쳐 leaf 노드(Plain Text, Start처럼 이름에 배열 대괄호 `[]`가 없는 단일 값)까지 내려가 선택하는 습관이 필요하다.

**적용 범위**: 미션명·날짜를 사용한 4곳(Sheets 분기 A/B, Discord 분기 A/B) 모두 동일 수정 필요 — 진행 상황은 `requirements-checklist.md`에서 추적.

## 9. 요구사항 원문 최종 대조에서 발견한 미비 사항 — 자동 실행 스케줄 누락

과제 원문 제약사항에 "프로젝트 2는 Trigger 발생 시 자동 실행되어야 한다"는 조건이 있는데, 그동안 모든 테스트를 Make의 **"Run once"(수동 실행)** 버튼으로만 진행해 화면 하단의 **"Every 15 minutes"(자동 스케줄) 토글이 계속 꺼진 상태**였다. 즉 실제로는 Notion에 새 항목을 추가해도 Make가 스스로 감지해 실행하는 게 아니라, 사람이 Make에 들어가 버튼을 눌러야만 동작하는 상태였다.

**수정**: 최종 점검 단계에서 발견해 "Every 15 minutes" 토글을 켬(`freetopic/make/16-auto-schedule-enabled.png`) → 이후로는 Notion 새 항목 추가가 곧 Trigger가 되어 자동 실행됨.

**학습 포인트**: 기능 구현과 "제약 조건 충족"은 별개로 확인해야 한다. 워크플로우 로직 자체는 처음부터 정상이었지만, "자동으로 실행되는가"라는 운영 조건은 별도로 점검하지 않으면 놓치기 쉽다 — 산출물 완성 후에도 원문 제약사항을 한 줄씩 다시 대조하는 과정에서 발견됨.

## 8. 두 번째 이슈: Discord 분기 A(보너스 O) 메시지가 채널에 안 보임 (시행착오 기록)

**문제**: 위 JSON 노출 버그를 고친 뒤 재실행했더니, Sheets는 두 분기 다 정상, Discord도 **X분기(✅ 미션 완료)는 채널에 정상 도착**했는데 **O분기(🎉 보너스까지 완료)만 채널에 안 보임**. 그런데 Make 실행 로그는 O분기 Discord 모듈도 초록 체크(성공)로 표시됨.

**원인 추정 및 확인 과정**:
1. Make의 Discord 모듈 Output을 열어보니 **실제 Message ID가 발급되어 있었음** — Discord API 호출 자체는 성공했다는 뜻. "아예 실행 안 됨"이 아니라 "다른 곳에 감"이라는 가설로 전환.
2. O분기(8번 모듈)와 X분기(9번 모듈)의 Output에서 각각 **Channel ID**를 직접 비교 → `...898`(O분기) vs `...897`(X분기)로 **서로 다른 채널**이었음이 확인됨.
3. Discord 서버에 **텍스트 채널 `#일반`과 음성 채널 `🔊 일반`이 동시에 존재**하는데, Make의 Channel ID 드롭다운에는 아이콘 구분 없이 "일반"이라는 이름만 표시되어, O분기 모듈 설정 시 실수로 음성 채널 쪽을 선택했던 것으로 확인됨. Discord API는 음성 채널에도 메시지 전송 자체는 받아주기 때문에 Make 로그는 정상 성공으로 표시되고, 정작 사용자가 보고 있던 텍스트 채널(`#일반`)에는 나타나지 않았음.

**수정**: O분기 Discord 모듈의 Channel ID를 X분기와 동일한 채널(텍스트 채널 `#일반`, `...897`)로 재선택 → 재실행 후 채널에 정상 도착 확인.

**학습 포인트**: Make 캔버스의 초록 체크(성공 표시)는 "해당 앱의 API 호출이 에러 없이 완료됐다"는 뜻일 뿐, "내가 의도한 결과가 나왔다"는 보장은 아니다. 특히 이번처럼 **같은 이름의 채널/리소스가 여러 개 있을 때는 드롭다운에서 이름만 보고 고르면 잘못된 대상을 선택하기 쉽다** — 이럴 땐 Channel ID 같은 고유 식별자를 직접 비교해서 확인해야 한다. 로그가 "성공"이어도 실제 결과물(채널 화면, 시트 값 등)을 눈으로 직접 확인하는 습관이 중요하다는 걸 두 번째 버그에서도 재확인.
